## Context
This session introduced explicit C# coding conventions for the OMG backend and aligned the codebase with a new async/await policy. The focus was to stop routinely using `.ConfigureAwait(false)` in ASP.NET Core backend code and to document language-level expectations for future contributors.

## What was done
- Added a `csharp-coding` rule file under `.cursor/rules` based on cursor.directory’s .NET development rules, tailored to OMG’s minimal API and DDD architecture.
- Documented an `Async & ConfigureAwait` policy that discourages `.ConfigureAwait(false)` in this backend, including a concrete before/after example for guidance.
- Searched the solution for all usages of `ConfigureAwait(false)` across tests, infrastructure, and API projects and confirmed they are standard backend async flows.
- Replaced `await ...ConfigureAwait(false)` with simple `await` in telemetrics, management, auth infrastructure, and API tests to match the new rule.
- Ran `dotnet test` for the full solution, confirming all tests (domain and API) still pass after the async style change.
- Committed and pushed the changes on `main` as `506b5566…` with the message “chore(csharp): add coding rules and drop ConfigureAwait”.

## Design decisions
- Chose to document C# conventions as a `.cursor/rules` file instead of a skill to keep project-wide style guidance lightweight and always-on for agents.
- Decided that `.ConfigureAwait(false)` should not be used in this ASP.NET Core backend by default, reserving it only for hypothetical reusable libraries that must run under UI or legacy synchronization contexts.
- Limited the scope of edits to mechanical removal of `.ConfigureAwait(false)` to avoid mixing behavioral changes with style cleanup in a single commit.
- Accepted the existing Result API warning in telemetrics domain for now, deferring a separate design discussion about result type inheritance and method hiding.
