## Context
This session focused on hardening the OMG API’s security posture based on a targeted review of authentication, telemetry ingestion, health checks, and error handling. The goal was to implement low-effort changes with high impact on confidentiality, integrity, and resilience.

## What was done
- Tightened `/api/v1/health` so dependency failures are logged server-side while non-development environments receive only generic ProblemDetails responses.
- Changed `/api/v1/auth/register` to return verification codes only in the Testing environment, reducing the risk of verification-code leakage in production.
- Introduced a dedicated `telemetry` rate-limiting policy keyed by IP and `X-Garden-Telemetry-Key`, and applied it to `POST /api/v1/telemetry/readings`.
- Documented how garden-level telemetry API keys are stored and added a concrete SQL-based rotation procedure in `telemetry-api-key-rotation.md`.
- Added global exception handling for non-development environments to log unhandled exceptions and return a generic 500 ProblemDetails response without stack traces.
- Enhanced refresh-token security by logging suspected reuse/invalid tokens and revoking all active refresh tokens for the affected user.
- Ran the full `dotnet test` suite (domain, infrastructure, and API tests), confirming all tests pass after these changes.

## Design decisions
- Chose environment-aware behavior (Development vs non-Development) for both health error details and global exception responses to preserve DX while reducing production information leakage.
- Kept the telemetry rate limiter configurable via `RateLimiting:Telemetry` in configuration to allow tuning limits per environment without code changes.
- Opted for a conservative refresh-token misuse response that logs and revokes active tokens, balancing security with implementation simplicity.
- Implemented telemetry key rotation initially as an operational runbook (SQL-based) rather than a public API surface to avoid expanding the external attack surface prematurely.
