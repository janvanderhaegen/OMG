## Context
Aligned the authentication registration flow with the existing event-driven architecture by introducing a dedicated Auth integration event publisher and ensuring registration emits a `SendRegistrationEmail` integration message. Strengthened tests around registration to verify that events are published without requiring a live RabbitMQ broker in the test environment.

## What was done
- Added `IAuthIntegrationEventPublisher` and `AuthIntegrationEventPublisher` in `OMG.Auth.Infrastructure` to wrap MassTransit `IPublishEndpoint` and publish `SendRegistrationEmail` messages.
- Registered the new Auth integration publisher in DI within `Program.cs`, alongside the existing `IGardenIntegrationEventPublisher`.
- Refactored the `/api/v1/auth/register` endpoint in `AuthEndpoints` to depend on `IAuthIntegrationEventPublisher` instead of using `IPublishEndpoint` directly.
- Extended the test `AuthEndpointTests.Register_Creates_User_Pending_Verification` to assert that a `SendRegistrationEmail` message is emitted with the expected user id, email, and verification code.
- Introduced a `CapturingAuthIntegrationEventPublisher` in `ManagementApiFactory` to capture published registration email messages during tests without starting MassTransit or RabbitMQ.
- Ran the full `dotnet test` suite to confirm all domain and API tests (including the updated auth tests) pass successfully.

## Design decisions
- Kept MassTransit as the single bus abstraction but funneled Auth messages through a context-specific publisher (`IAuthIntegrationEventPublisher`), mirroring the pattern already used for garden integration events.
- In tests, preferred a capturing publisher over standing up a real bus and consumer, focusing on observable behavior (correct integration message payload) while keeping the test environment lightweight and deterministic.
