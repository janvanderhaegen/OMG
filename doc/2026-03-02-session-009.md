## Context
Implemented safer account deletion in the Auth bounded context by anonymizing user data and shifting garden cleanup to an event-driven flow. Extended messaging so that other bounded contexts can react to user removal without tight coupling to the Auth HTTP surface.

## What was done
- Updated the `DeleteAccount` auth endpoint to anonymize personal data (email, username, phone, verification and confirmation flags) before marking the user as deleted and revoking refresh tokens.
- Introduced a new `UserRemoved` integration contract in the Auth messaging project and extended `AuthIntegrationEventPublisher` to emit this event when an account is deleted.
- Added a `UserRemovedConsumer` in Management infrastructure that reacts to `UserRemoved`, soft-deletes all gardens for that user via the domain model, and publishes the corresponding garden integration events.
- Registered the new consumer in the MassTransit `MessagingConfiguration` so it participates in the existing outbox-driven messaging pipeline.
- Updated API tests to assert anonymization behavior, token revocation, and `UserRemoved` message publication, and extended the test `ManagementApiFactory` to capture `UserRemoved` messages.
- Committed and pushed the changes to `main` (`feat(auth): anonymize account deletion and emit user-removed events`).
