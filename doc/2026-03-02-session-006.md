## Context
Refined the garden and plant management API endpoints and tests to enforce the canonical DDD workflow and to ensure that tests assert both HTTP responses and the actual persisted state.
This session focused on endpoint ordering around integration events vs. unit-of-work commits, plus tightening integration tests so they catch persistence regressions.

## What was done
- Updated the Garden Management architecture section in `ARCHITECTURE-FINAL.md` to specify that endpoints must publish integration events, then call `IManagementUnitOfWork.SaveChangesAsync(...)` as the final infrastructural step before mapping responses.
- Adjusted `ManagementGardenEndpoints` and `ManagementPlantEndpoints` so create/update/delete handlers follow the documented workflow: operate on aggregates, persist via `IGardenRepository`, publish integration events, then commit once at the end.
- Strengthened `GardenManagementEndpointTests` and `PlantManagementEndpointTests` so create/update scenarios now verify the stored `GardenEntity`/`PlantEntity` rows in addition to the returned DTOs.
- Committed these changes on branch `main` as `ca6e7aacff70a09ce8776c654692308c2fe49b18` and pushed to `origin/main`.

## Design decisions
- Treat the combination of repository persistence + integration event publishing + unit-of-work commit as a single, well-ordered workflow that is consistent across all management endpoints.
- Require API tests to validate both user-visible behavior (responses) and underlying persistence, to prevent silent failures where aggregates are updated in memory but not stored correctly.
